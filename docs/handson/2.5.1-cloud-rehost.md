# 2.5.1. Rehost（Spoke1）

Azure Migrate の **Migration and modernization**（エージェントベース方式）を使い、`APP01` と `DB01` を Azure VM に Lift & Shift 移行します。

## 目的

- Azure Migrate Server Migration（物理サーバー / エージェントベース）を体験する
- レプリケーションアプライアンスを構成し、継続的レプリケーションを行う
- テスト移行 → カットオーバーの一連のフローを理解する

## 前提条件

- [`2.4-cloud-assessment.md`](./2.4-cloud-assessment.md) の評価が完了している
- `rg-spoke1`・`vnet-spoke1`（snet-web / snet-db）が作成されている
- Hub-Spoke VNet ピアリングが接続済みである
- ServerSubnet に NAT Gateway が付与されている（アプライアンスのインターネット接続用）

## 移行先構成

| コンポーネント | 移行元 | 移行先 |
|---|---|---|
| Web サーバー | APP01（vm-onprem-web / 10.0.1.6） | vm-spoke1-web（snet-web） |
| DB サーバー | DB01（vm-onprem-sql / 10.0.1.5） | vm-spoke1-sql（snet-db） |

## 特徴

- **メリット**: アプリやOS への変更が不要。最も速い移行パターン
- **デメリット**: VM 運用が継続するため、運用負荷は高いまま

<details>
<summary>（任意）PsPing のインストール</summary>

移行中のサービス断を確認するため、DC01 に [PsPing](https://learn.microsoft.com/ja-jp/sysinternals/downloads/psping?wt.mc_id=MVP_479930) をインストールします。PsPing は TCP ポート単位の疎通確認ができるため、Windows Firewall で ICMP がブロックされていても使えます。

### インストール手順

1. Azure Bastion 経由で **DC01**（vm-onprem-ad / 10.0.1.4）に RDP 接続
2. 管理者 PowerShell で以下を実行:

   ```powershell
   # PSTools をダウンロード・展開
   [Net.ServicePointManager]::SecurityProtocol = [Net.SecurityProtocolType]::Tls12
   Invoke-WebRequest -Uri "https://download.sysinternals.com/files/PSTools.zip" -OutFile "$env:TEMP\PSTools.zip" -UseBasicParsing
   Expand-Archive -Path "$env:TEMP\PSTools.zip" -DestinationPath "C:\Tools" -Force

   # PATH に追加
   $env:Path += ";C:\Tools"
   [Environment]::SetEnvironmentVariable("Path", "$([Environment]::GetEnvironmentVariable('Path','Machine'));C:\Tools", "Machine")
   ```

### 使い方

移行テスト時に DC01 のコマンドプロンプトまたは PowerShell で実行します:

```cmd
:: APP01 の HTTP (ポート 80) を継続監視（Ctrl+C で停止）
psping64 -accepteula -t 10.0.1.6:80

:: DB01 の SQL Server (ポート 1433) を継続監視
psping64 -accepteula -t 10.0.1.5:1433
```

移行中にサービスが停止すると応答が途切れるため、ダウンタイムの開始・終了時刻を確認できます。

> **補足**: PsPing は [Sysinternals PSTools](https://learn.microsoft.com/ja-jp/sysinternals/downloads/psping) に含まれるスタンドアロン exe で、コマンドプロンプト・PowerShell の両方で使用可能です。

<details>
<summary>Step 1: レプリケーションアプライアンス VM の作成</summary>

レプリケーションアプライアンスは、移行元サーバーからのデータを受け取り Azure へ転送する中継役です。検出用アプライアンス（vm-onprem-migrate）とは別の VM が必要です。

### 1-1. VM の作成

Azure Portal または CLI で、オンプレ VNet の ServerSubnet にアプライアンス用 VM を作成します。

| 項目 | 値 |
| --- | --- |
| リソース グループ | rg-onprem |
| 仮想マシン名 | vm-onprem-repl |
| コンピュータ名 | REPL |
| リージョン | 東日本 |
| イメージ | Windows Server 2022 Datacenter |
| サイズ | Standard_D16s_v3（16 vCPU / 64 GB） |
| 管理者ユーザー名 | labadmin |
| 管理者パスワード | オンプレ環境と同じパスワード |
| パブリック IP | なし |
| 仮想ネットワーク | vnet-onprem |
| サブネット | ServerSubnet (10.0.1.0/24) |
| NIC DNS サーバー | 168.63.129.16（Azure DNS 直指定） |
| OS ディスクサイズ | **1024 GB（1 TB）** |

> **重要**: レプリケーションアプライアンスのインストーラー（DRInstaller.ps1）は **C: ドライブに 600 GB 以上の空き容量** を要求します。既定の 128 GB では不足するため、OS ディスクを 1 TB に設定してください。

> **ポイント**: VNet の DNS はオンプレ AD（10.0.1.4）を向いていますが、アプライアンスは Azure サービスの名前解決が必要なため、NIC レベルで Azure DNS を直接設定します。

<details>
<summary>CLI で作成する場合</summary>

```bash
# NIC 作成（Azure DNS 直指定）
az network nic create -g rg-onprem -n nic-vm-onprem-repl \
  --vnet-name vnet-onprem --subnet ServerSubnet \
  --dns-servers 168.63.129.16 -o none

# VM 作成
az vm create -g rg-onprem -n vm-onprem-repl \
  --computer-name REPL \
  --image MicrosoftWindowsServer:WindowsServer:2022-datacenter-g2:latest \
  --size Standard_D16s_v3 \
  --nics nic-vm-onprem-repl \
  --admin-username labadmin \
  --admin-password '<パスワード>' \
  --public-ip-address "" \
  --os-disk-name osdisk-vm-onprem-repl \
  --os-disk-size-gb 1024 \
  -o none
```

</details>

<details>
<summary>Step 3: レプリケーションアプライアンスのセットアップ</summary>

> **参考**: [Deploy Azure Site Recovery replication appliance - Modernized](https://learn.microsoft.com/en-us/azure/site-recovery/deploy-vmware-azure-replication-appliance-modernized#set-up-the-appliance-through-powershell)

### 3-1. インストーラーのダウンロードと実行

1. Azure Bastion 経由で **vm-onprem-repl** に RDP 接続する
2. Step 2 の画面で **[ステップ 1: アプライアンスを作成する]** のリンクからインストーラー（PowerShell zip）をダウンロード
   - 直接ダウンロード URL: `https://aka.ms/V2ARcmApplianceCreationPowershellZip`
3. ダウンロードした zip を展開する
4. 管理者 PowerShell で展開先フォルダに移動し、以下を実行:

   ```powershell
   .\DRInstaller.ps1
   ```

5. インストールが完了すると、**Microsoft Azure Appliance Configuration Manager** が自動的にブラウザで開く

### 3-2. 前提条件の確認

Configuration Manager が起動すると、以下の前提条件が自動チェックされます:

- インターネット接続
- 時刻同期
- システム構成
- グループポリシー（レジストリアクセス、コマンドプロンプト、PowerShell 実行ポリシー等）

すべて緑チェックになったら次へ進みます。

> **トラブルシュート**: インターネット接続チェックが失敗する場合は、ServerSubnet の NAT Gateway と NIC の DNS 設定（168.63.129.16）を確認してください。

### 3-3. アプライアンスの登録

1. **アプライアンスの接続オプション**（FQDN または NAT IP）を選択 → **[Continue]**
2. Azure Portal の Step 2 に戻り、**[キーの生成]** をクリックしてキーをコピー
3. Configuration Manager に戻り:
   - **Friendly name**: `repl-appliance`（任意の識別名）
   - **Azure Site Recovery replication appliance key**: コピーしたキーを貼り付け
4. **[Login]** をクリック → デバイスコード認証で Azure にログイン
5. 登録成功後、サブスクリプション / リソースグループ / Recovery Services Vault の情報が表示される
6. **[Continue]** をクリック

### 3-4. 物理サーバーの資格情報の登録

> **補足: Azure Arc への自動登録**
> アプライアンス登録時に、アプライアンスVM が **Azure Arc-enabled server** として自動的に登録されます（例: `appliance02995fa9`）。これは Modernized アーキテクチャの仕様で、Arc 経由でアプライアンスの正常性監視やポリシー適用が行われます。アプライアンス使用中は Arc 登録を解除しないでください。

1. **[Provide Physical server details]** を展開
2. **[Add credentials]** をクリック:

   | 項目 | 値 |
   | --- | --- |
   | OS | Windows |
   | フレンドリ名 | onpremadmin |
   | ユーザー名 | lab\labadmin |
   | パスワード | オンプレ環境の管理者パスワード |

3. **[Add]** をクリック
4. **[Add server]** をクリックし、移行対象サーバーを追加:

   | IP アドレス | サーバー | 資格情報 |
   | --- | --- | --- |
   | 10.0.1.5 | DB01（SQL Server） | onpremadmin |
   | 10.0.1.6 | APP01（IIS / Web アプリ） | onpremadmin |

5. **[Continue]** をクリック → コンポーネントのインストールと登録が開始される（最大 30 分）

> **注意**: ブラウザを閉じずに構成完了まで待機してください。

### 3-5. アプライアンスの接続確認

Azure Portal の Azure Migrate プロジェクトに戻り、**[レプリケーション アプライアンス]** にアプライアンスが表示されていることを確認します。

> アプライアンスが接続されるまで約 15 分かかる場合があります。

<details>
<summary>Step 4: レプリケーションの有効化</summary>

アプライアンスが接続されたら、移行対象のレプリケーションを設定します。ターゲットのサブネットはレプリケーション設定ごとに 1 つしか選択できないため、APP01（snet-web）と DB01（snet-db）を **別々に** 設定します。

### 4-0. ソース VM の前提条件設定

レプリケーション有効化時に、アプライアンスからソース VM へモビリティサービスがプッシュインストールされます。これには Admin 共有・WMI・特定ポートへのアクセスが必要です。

**APP01（vm-onprem-web）と DB01（vm-onprem-sql）のそれぞれ** に Bastion 接続し、管理者 PowerShell で以下を実行:

```powershell
# File and Printer Sharing を有効化（Admin 共有経由のファイルコピーに必要）
Set-NetFirewallRule -DisplayGroup "File and Printer Sharing" -Enabled True

# WMI を有効化（リモートでのサービスインストール・起動に必要）
Set-NetFirewallRule -DisplayGroup "Windows Management Instrumentation (WMI)" -Enabled True

# TCP 443/9443 を開放（モビリティサービスとアプライアンス間の通信に必要）
New-NetFirewallRule -DisplayName "ASR TCP 443" -Direction Inbound -Protocol TCP -LocalPort 443 -Action Allow
New-NetFirewallRule -DisplayName "ASR TCP 9443" -Direction Inbound -Protocol TCP -LocalPort 9443 -Action Allow

# RemoteRegistry サービスを開始（レジストリ情報のリモート取得に必要）
Set-Service RemoteRegistry -StartupType Automatic
Start-Service RemoteRegistry
```

> **重要**: この設定を行わないと、レプリケーション有効化時にエラー 322008（Connectivity failure）が発生し、モビリティサービスのインストールに失敗します。

### 4-1. APP01 のレプリケーション

1. Azure Migrate → プロジェクト → **[Migration and modernization]** → **[レプリケート]** をクリック
2. **[ソースの設定]**:

   | 項目 | 値 |
   | --- | --- |
   | マシンは仮想化されていますか？ | 物理またはその他 |
   | オンプレミス アプライアンス | repl-appliance |

3. **[仮想マシン]** タブで **APP01** を選択
4. **[ターゲットの設定]**:

   | 項目 | 値 |
   | --- | --- |
   | サブスクリプション | 使用中のサブスクリプション |
   | リソースグループ | rg-spoke1 |
   | レプリケーション ストレージ アカウント | （自動作成または選択） |
   | 仮想ネットワーク | vnet-spoke1 |
   | サブネット | **snet-web** |

5. **[コンピューティング]** タブ:

   | 移行元 | VM 名 | サイズ |
   | --- | --- | --- |
   | APP01 | vm-spoke1-web | 評価結果の推奨サイズ |

6. **[ディスク]** タブでレプリケート対象のディスクを確認
7. **[レプリケーションの確認と開始]** → **[レプリケート]** をクリック

### 4-2. DB01 のレプリケーション

同じ手順で DB01 用のレプリケーションを追加します。

1. **[Migration and modernization]** → **[レプリケート]** をクリック
2. **[ソースの設定]**: APP01 と同じ（物理またはその他 / repl-appliance）
3. **[仮想マシン]** タブで **DB01** を選択
4. **[ターゲットの設定]**:

   | 項目 | 値 |
   | --- | --- |
   | サブスクリプション | 使用中のサブスクリプション |
   | リソースグループ | rg-spoke1 |
   | レプリケーション ストレージ アカウント | （APP01 と同じものを選択） |
   | 仮想ネットワーク | vnet-spoke1 |
   | サブネット | **snet-db** |

5. **[コンピューティング]** タブ:

   | 移行元 | VM 名 | サイズ |
   | --- | --- | --- |
   | DB01 | vm-spoke1-sql | 評価結果の推奨サイズ |

6. **[ディスク]** タブでレプリケート対象のディスクを確認
7. **[レプリケーションの確認と開始]** → **[レプリケート]** をクリック

> **ポイント**: レプリケーションが開始されると、移行対象 VM にモビリティサービスエージェントが自動的にプッシュインストールされます。初回の完全レプリケーションには環境に応じて数時間かかる場合があります。

### レプリケーション状態の確認

| ステータス | 説明 |
| --- | --- |

| 保護済み | 差分レプリケーションが継続中 — テスト移行が可能 |
| 重大 | エラーが発生、対処が必要 |

**[保護済み]** になったら次のステップへ進みます。

テスト移行で問題がなければ、本番カットオーバーを実行します。

- DNS レコードの更新（必要に応じて）

<details>
<summary>トラブルシューティング</summary>

### モビリティサービスのプッシュインストールが失敗する（エラー 322008）

レプリケーション有効化時に `Connectivity failure - Mobility service installation failed` が表示される場合、レプリケーションアプライアンスからソース VM への接続要件が満たされていません。

**原因**: モビリティサービスのプッシュインストールでは、アプライアンスからソース VM への Admin 共有・WMI・特定ポートへのアクセスが必要です。

**対処方法**: 対象 VM（APP01 / DB01）それぞれの管理者 PowerShell で以下を実行:

```powershell
# File and Printer Sharing を有効化（Admin 共有 C$/ADMIN$ 経由のファイルコピーに必要）
Set-NetFirewallRule -DisplayGroup "File and Printer Sharing" -Enabled True

# WMI を有効化（リモートでのサービスインストール・起動に必要）
Set-NetFirewallRule -DisplayGroup "Windows Management Instrumentation (WMI)" -Enabled True

# TCP 443/9443 を開放（モビリティサービスとアプライアンス間の通信に必要）
New-NetFirewallRule -DisplayName "ASR TCP 443" -Direction Inbound -Protocol TCP -LocalPort 443 -Action Allow
New-NetFirewallRule -DisplayName "ASR TCP 9443" -Direction Inbound -Protocol TCP -LocalPort 9443 -Action Allow

# RemoteRegistry サービスを開始（レジストリ情報のリモート取得に必要）
Set-Service RemoteRegistry -StartupType Automatic
Start-Service RemoteRegistry
```

設定後、Azure Portal のレプリケーション詳細画面で **[ジョブの再起動]** をクリックしてリトライします。

| 要件 | 目的 |
| --- | --- |
| File and Printer Sharing | Admin 共有（`C$`、`ADMIN$`）経由でモビリティサービス exe をコピー |
| WMI | リモートでサービスのインストール・起動を実行 |
| TCP 443 / 9443 | モビリティサービスとアプライアンス間の通信チャネル |
| RemoteRegistry | レジストリ情報のリモート取得 |

### レプリケーションが進まない

- レプリケーションアプライアンス（vm-onprem-repl）のインターネット接続を確認
- NAT Gateway が ServerSubnet に付与されているか確認
- Azure Portal → レプリケーション詳細画面でエラーメッセージを確認

### テスト移行後の VM にアクセスできない

- NSG ルールで Bastion からの RDP（3389）が許可されているか確認
- テスト VM が正しい VNet / サブネットに接続されているか確認

</details>
